PitCrew supports internationalization (i18n) to display your site in multiple languages. This guide explains how to configure the site language and add support for new languages.

## Supported Languages

Out of the box, PitCrew supports these languages:

| Code | Language |
|------|----------|
| `en` | English (default) |
| `fr` | French (Fran&ccedil;ais) |
| `de` | German (Deutsch) |
| `es` | Spanish (Espa&ntilde;ol) |
| `zh` | Chinese Simplified (&nbsp;中文&nbsp;) |
| `ja` | Japanese (&nbsp;日本語&nbsp;) |

## Configuring the Site Language

To change your site's language, edit the `lang` setting in `_config.yml`:

```yaml
# Language / Internationalization (i18n)
# Supported languages: en, fr, de, es, zh, ja
lang: "fr"  # Change to your preferred language code
```

After changing the language, rebuild your site for the changes to take effect.

## How It Works

Translation files are stored in `_data/i18n/` as YAML files:

```
_data/
  i18n/
    en.yml    # English (default)
    fr.yml    # French
    de.yml    # German
    es.yml    # Spanish
    zh.yml    # Chinese
    ja.yml    # Japanese
```

Each file contains nested keys for different parts of the site:

```yaml
nav:
  home: "Home"
  about: "About"
  season: "Season"

footer:
  quick_links: "Quick Links"
  about_us: "About Us"
```

Templates use the `i18n.html` include to fetch translated strings:

```liquid
{% raw %}{% include i18n.html key="nav.home" %}{% endraw %}
```

## Adding a New Language

### Step 1: Create the Translation File

Create a new YAML file in `_data/i18n/` with your language code (e.g., `pt.yml` for Portuguese):

```bash
cp _data/i18n/en.yml _data/i18n/pt.yml
```

### Step 2: Translate the Strings

Open the new file and translate each string value. Keep the keys the same:

```yaml
# Portuguese - pt.yml
nav:
  home: "In&iacute;cio"
  about: "Sobre"
  season: "Temporada"
  awards: "Pr&ecirc;mios"
  sponsors: "Patrocinadores"
  docs: "Documenta&ccedil;&atilde;o"
  news: "Not&iacute;cias"
  menu: "Menu"
  skip_to_content: "Pular para o conte&uacute;do principal"
```

### Step 3: Update the Config

Set your new language code in `_config.yml`:

```yaml
lang: "pt"
```

### Step 4: Rebuild the Site

Run Jekyll to rebuild with the new language:

```bash
bundle exec jekyll serve
```

## Fallback Behavior

If a translation key is missing in the current language file, the system automatically falls back to English (`en.yml`). This ensures your site always displays text even if translations are incomplete.

## Using Translations in Templates

### Basic Usage

```liquid
{% raw %}{% include i18n.html key="nav.home" %}{% endraw %}
```

### With Fallback Text

```liquid
{% raw %}{% include i18n.html key="custom.key" fallback="Default Text" %}{% endraw %}
```

### In Attributes

```liquid
{% raw %}<a href="/" aria-label="{% include i18n.html key="nav.home" %}">{% endraw %}
```

## Best Practices

1. **Keep keys consistent**: Never change existing keys as this will break translations
2. **Use nested structure**: Group related strings under common prefixes
3. **Provide context**: Use descriptive key names that indicate where the string is used
4. **Test thoroughly**: Check all pages after adding translations
5. **Start with English**: Always add new strings to `en.yml` first, then translate

## Contributing Translations

Please ensure all strings are translated before submitting.

## Troubleshooting

### Translations Not Showing

1. Verify the `lang` setting in `_config.yml`
2. Check that the translation file exists in `_data/i18n/`
3. Ensure the YAML syntax is valid (use a YAML validator)
4. Rebuild the site after making changes

### Special Characters

For special characters, use HTML entities or ensure your file is saved as UTF-8:

```yaml
# Using HTML entities
title: "Caf&eacute;"

# Or use UTF-8 directly
title: "Caf&eacute;"
```

### Missing Keys

If you see a key path displayed instead of text (e.g., `nav.home`), the key is missing from both the current language and English fallback files.
